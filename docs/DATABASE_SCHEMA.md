# Live database compatibility report

Generated by inspecting the live Supabase project (`zglbgqeccebqnijcqfkb`)
directly via `list_tables`, `information_schema`, `pg_constraint`,
`pg_trigger`, `pg_proc`, `pg_policies`, and the realtime publication —
not from any local migration file, per this rebuild's instruction that
the database is authoritative and the repo may not contain a faithful
baseline. `src/shared/database.types.ts` is generated straight from this
project and should be regenerated (`generate_typescript_types`) instead
of hand-edited whenever the schema changes.

## 1. Tables verified (16)

`organizations`, `organization_members`, `organization_entitlements`,
`locations`, `menus`, `menu_sections`, `menu_items`, `presentation_assets`,
`scenes`, `display_layouts`, `display_layout_tiles`, `screens`,

Every table carries `org_id`; most also carry `location_id` and use
composite foreign keys like `(id, org_id, location_id)` so a child row's
location can never silently drift from its parent's.

## 2. Business logic already enforced at the database layer

This schema does far more than earlier revisions — a lot of what the
task brief calls "application requirements" is already a trigger or
constraint, not something the service layer needs to reimplement:

| Rule | Where |
|---|---|
| Screen must be `status='ready'` to join a session | `prepare_session_screen_assignment()` |
| Entitlement screen-per-session limit (personal=1, plus=5, pro≥10) | `prepare_session_screen_assignment()`, counts non-removed rows in the *same session* |
| `independent`/`single` mode requires every enabled screen to have its own `layout_id` | `prepare_session_screen_assignment()` (on add) + `validate_display_session_activation()` (on activate) |
| `single` mode allows exactly one enabled screen | both of the above |
| `duplicate`/`extend` require `shared_layout_id`; `independent`/`single` forbid it | `display_sessions_check` |
| A session must be created as `draft` | `validate_new_display_session()` |
| Activating a session needs ≥1 enabled screen | `validate_display_session_activation()` |
| One screen can only be `active` in one session at a time | `one_live_session_per_screen_idx` (partial unique index) |
| One primary screen per session | `one_primary_screen_per_session_idx` |
| Stopping a session releases all its screens (history preserved) | `handle_display_session_status()` |
| Tile geometry stays inside the canvas (0–100%, non-overlapping edges) | `display_layout_tiles_check`/`check1` + percent checks |
| Pairing code auto-expires ≤30 minutes from creation | `screen_pairing_codes_check1` |
| A `screen` row's `org_id`/`location_id`/`claimed_at`/`revoked_at` all move together with `status` | `screens_check` |
| Scene content matches its declared type (`menu`→`menu_id`, `powerpoint`→`presentation_asset_id`, never both) | `scenes_check` |
| `external_identities` mapping is immutable once linked | `forbid_external_identity_relink()` trigger |

The service layer (`src/domain/*`) therefore validates *shapes* client-side
for fast feedback, then lets these triggers be the actual authority —
`mapPostgresError` (`src/shared/errors.ts`) translates their raised
messages into the stable API error contract. Three of these mappings were
verified against real trigger output in rolled-back transactions:
`"A display session must be created as a draft before it can be started"`
→ `SESSION_NOT_DRAFT`, `"This plan allows at most 1 screen(s) per
session"` → `SCREEN_LIMIT_REACHED`, and the `display_sessions_check`
constraint name → `LAYOUT_INVALID`.

## 4. Differences from the task's assumed schema

- **No `display_automations` failure-tracking columns.** The table has
  `last_run_at` and `next_run_at` but no `last_success_at`, `last_error`,
  or `failure_count`. `automations-run` currently stamps `last_run_at` on
  both success and failure (the only durable signal available today) and
  logs failures to the function's own logs. See §5 for the proposed,
  **unapplied** fix.
- **No storage bucket for presentations.** `presentation_assets` has
  `lxc_source_key` / `lxc_manifest_key`, not a Supabase Storage path —
  confirming presentations live on the private LXC service, not in this
  project. `presentation-upload` was rewritten around that contract (see
  `docs/API.md`).
- **Manager-role membership writes are owner+admin**, not owner-only.
  `organization_members_manage_admin` allows `role IN ('owner','admin')`.
  The task brief says "owner-only" for membership operations; the live
  RLS is what's authoritative, so `src/domain/organizations.ts` does not
  add a stricter client-side gate than the database already enforces.

## 5. Proposed, unapplied changes — full review

Per the "database mismatch procedure" for this run: found, documented,

### 5a. Revoke stray grants on `screens` / `screen_pairing_codes` — security priority

**Exact SQL:**

```sql
revoke all on table public.screens from anon, authenticated;
revoke all on table public.screen_pairing_codes from anon, authenticated;

grant select (id, org_id, location_id, name, status, claimed_at, last_seen_at, revoked_at, created_at, updated_at)
  on public.screens to authenticated;
grant update (name, location_id)
  on public.screens to authenticated;
-- screen_pairing_codes gets no client grant at all — service-role only,
-- matching its current zero-policy RLS.
```

**Rollback SQL:**

```sql
revoke select (id, org_id, location_id, name, status, claimed_at, last_seen_at, revoked_at, created_at, updated_at) on public.screens from authenticated;
revoke update (name, location_id) on public.screens from authenticated;
grant all on table public.screens to anon, authenticated;
grant all on table public.screen_pairing_codes to anon, authenticated;
```

**Why needed:** `anon` and `authenticated` currently hold full
table-level `SELECT`/`INSERT`/`UPDATE`/`DELETE` on both tables —
including `screens.device_token_hash` and
`screen_pairing_codes.code_hash` — left over from Supabase's default
`GRANT ALL ... TO anon, authenticated`, confirmed via
`information_schema.column_privileges` (not assumed). Found again, not
newly introduced by this rebuild.

**Does existing RLS already mitigate it?** Partially, and that's exactly
why this is fragile rather than actively broken:
- `screens` has RLS restricting rows to `org_id IS NOT NULL AND
  is_org_member(org_id)`, so unclaimed/pairing rows (`org_id IS NULL`)
  are invisible regardless of the grant, and claimed rows are only
  visible to that org's own authenticated managers.
- `screen_pairing_codes` has RLS **enabled with zero policies**, which
  denies all access to every non-owner role outright — the grant is
  currently inert dead weight on this table specifically.

  Neither of these RLS facts means the grant is safe to leave: RLS
  policies are additive and can change independently of grants (a future
  migration adding *any* permissive policy to either table would
  instantly turn the existing over-broad grant into a live exposure of
  device-credential and pairing-code hashes, with no additional review
  step catching it). Defense-in-depth means the grant should already
  match the intent, not rely on RLS being the only backstop.

**Does application code work without it?** Yes, unconditionally. Every
Edge Function that touches either table (`screen-register`,
`screen-claim`, `screen-credential-rotate`, `display-gateway`) uses the
service-role client, which bypasses RLS and grants entirely. No
browser-side code (`src/domain/screens.ts`) ever selects
`device_token_hash` — it explicitly types its `Screen` shape as `Omit<...,
"device_token_hash">` and hardcodes a safe column list. Applying this
migration changes nothing observable in the running application.

**Locking/downtime risk:** None. `REVOKE`/`GRANT` on a table take an
`ACCESS EXCLUSIVE` lock only *briefly* to update the catalog entry — no
table rewrite, no row scan, sub-millisecond in practice on tables this
size (currently near-empty). Safe at any time, no maintenance window
needed.

**Recommended before initial production deployment?** **Yes.** This is
the one item in this report I'd block a first production rollout on: it
costs nothing (verified above — zero app-code impact, zero downtime) and
removes a real, if not-yet-exploited, exposure path for kiosk device
credentials and pairing-code hashes.

### 5b. `display_automations` failure/success tracking — optional

**Exact SQL:**

```sql
alter table public.display_automations
  add column last_success_at timestamptz,
  add column last_error text,
  add column failure_count integer not null default 0;
```

**Rollback SQL:**

```sql
alter table public.display_automations
  drop column last_success_at,
  drop column last_error,
  drop column failure_count;
```

**Why needed:** to distinguish "attempted" from "succeeded", capture the
actual failure reason per-automation, and let a worker circuit-break a
permanently-broken automation instead of retrying forever. Today
`automations-run` (`supabase/functions/automations-run/index.ts`) only
has `last_run_at`/`next_run_at` to work with — every attempt (success or
failure) advances `last_run_at` identically, and a failing one-shot
automation is retried on a fixed 5-minute backoff with no upper bound
(see `docs/SCHEDULER.md` § Retry behavior).

**Does existing RLS already mitigate it?** Not applicable — this isn't a
security gap, it's a missing observability column. No RLS interaction.

**Does application code work without it?** Yes, functionally — automations
still execute, still claim atomically, still retry. What's missing is
purely observability: there's no queryable "which automations are
currently failing" view, and no automatic circuit-breaker for a
persistently broken one. `automations-run` already has the call sites
ready to populate these columns (guarded so they're simply no-ops
today) — no application code changes would be needed if this migration
is applied, only uncommenting/wiring the three field-sets.

**Locking/downtime risk:** Low. `ALTER TABLE ... ADD COLUMN` with a
constant/`NULL` default (all three columns here — `failure_count`
defaults to a literal `0`, not an expression, and Postgres 11+ handles
constant defaults on `ADD COLUMN` without a table rewrite) takes a brief
`ACCESS EXCLUSIVE` lock for the catalog change only, not a full table
rewrite. Safe at any time given current table size; would be worth a
maintenance-window mention only if `display_automations` grows very
large in the future.

**Recommended before initial production deployment?** **Optional.** The
application functions correctly without it. Recommended before *relying
on automations in a way where silent failures would be costly* (e.g.
unattended overnight schedule changes) — not a blocker for an initial
deployment where a human can still notice a session that didn't start.
